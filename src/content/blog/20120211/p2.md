[s2jdbc](http://d.hatena.ne.jp/keyword/s2jdbc)で、JOIN数が多い場合の注意について。 プロジェクトも中盤、さあslow-query-logを元に[SQL](http://d.hatena.ne.jp/keyword/SQL)のパフォーマンスチューニングするか、という時、困った事が・・

```java
public class Test1Service extends AbstractService<Test1> {
    public List<Test1> find(Integer param1) {
        return select().leftOuterJoin(Test1Names.test2()).leftOuterJoin(Test1Names.test3())
            .where(new SimpleWhereEx().eq(Test1Names.CTinyint(), param1)).getResultList();
    }
}
```

test1を親に、test2、test3を外部結合しています。さて、どのようにクエリが発行されたでしょうか。

おおお・・・ JOINしたテーブルも含めた全テーブルの全カラムselectしてる・・・ 実はこの[s2jdbc](http://d.hatena.ne.jp/keyword/s2jdbc)ですが、現状正規のやり方でselectするカラムを選択できません。 このサンプルではJOIN数は2テーブルだけですが、5〜10テーブルJOINしたらどうなるでしょう。 [SQL](http://d.hatena.ne.jp/keyword/SQL)自体は一瞬で結果が返ります。 しかし、問題はそこではなかった・・・・

[SQL](http://d.hatena.ne.jp/keyword/SQL)の結果セットをEntityのListにマッピングする処理が遅い！！ よくあるのが「なんかこのservice遅いよ？」とDBAに相談が行くが、serviceで実行するのではなく、mysqlclientで[SQL](http://d.hatena.ne.jp/keyword/SQL)を直に叩くと速い。 結局entityへのマッピングが遅いのでした。

この件を踏まえると、 JOIN数が多いか、多くなりそうな場合は最初から[SQL](http://d.hatena.ne.jp/keyword/SQL)ファイルに手で[SQL](http://d.hatena.ne.jp/keyword/SQL)を書いた方がいい ということになりますね。

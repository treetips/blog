<a id="Google公式のWorkbox"></a>

### Google公式のWorkbox

[Workbox | Google Developers](https://developers.google.com/web/tools/workbox/)

[GoogleChrome/workbox](https://github.com/GoogleChrome/workbox)

実はGoogle、というよりGoogle Chromeが公式リポジトリにWorkboxというものをGithubで公開しており、これを使うと@angular/service-workerのような自動生成が可能になります。

ServiceWorker関連ファイルを自動生成する方法は、以下の3種類あります。

- cliで生成する
- node_moduleで生成する
- webpackで生成する（workbox-webpack-pluginでwebpack.config.jsに実装を記述する方式）

最初はwebpack(workbox-webpack-plugin)を使うやり方で試したのですが、webpackへの強い依存があると嫌ですし、Parcelが登場したりしているので、webpackから独立したcli方式が良いのではないかと思いました。

<a id="workbox-cliで自動生成する"></a>

### workbox cliで自動生成する

こちらが本命のcli形式です。

<a id="workbox-cliのインストール"></a>

#### workbox-cliのインストール

```sh
npm install workbox-cli --global
```

<a id="workbox-configjsを生成する"></a>

#### workbox-config.jsを生成する

workbox wizardを実行すると、npm initのような対話式ウィザードでworkbox-config.jsが生成できます。

```sh
tree:tree-maps-go tree$ workbox wizard
? What is the root of your web app (i.e. which directory do you deploy)? assets/
? Which file types would you like to precache? json, tsv, txt, xml, js, png, ico
? Where would you like your service worker file to be saved? assets/sw.js
? Where would you like to save these configuration options? workbox-config.js
To build your service worker, run

  workbox generateSW workbox-config.js

as part of a build process. See https://goo.gl/fdTQBf for details.
You can further customize your service worker by making changes to workbox-config.js. See https://goo.gl/gVo87N for details.
```

workbox-config.jsは、どこにあるファイルを、どれくらいの期間キャッシュするか、といった設定情報javascriptを自動生成するために使われます。

<a id="workbox-wizardで生成されたworkbox-configjs"></a>

##### workbox wizardで生成されたworkbox-config.js

```javascript
module.exports = {
  "globDirectory": "assets/",
  "globPatterns": [
    "**/*.{json,tsv,txt,xml,js,png,ico}"
  ],
  "swDest": "assets/sw.js"
};
```

workbox wizardの入力内容によって中身が変わります。

このファイルはnpm initによってpackage.jsonが生成されるのと同様に最初に雛形的に生成するものなので、これを任意に編集して拡張子毎に寿命設定を付けたり、apiのキャッシュ設定を追加したりします。今回はそこまでせずこの設定で試していきます。

<a id="generateSWしてswjsを自動生成する"></a>

#### generateSWしてsw.jsを自動生成する

```sh
workbox generateSW workbox-config.js
```

ターミナル上で上記コマンドで生成できますが、npm script化します。

```json
"scripts": {
  "build": "NODE_ENV=product webpack -p --mode production --progress --color && npm run build-workbox",
  "build-workbox": "workbox generateSW workbox-config.js"
}
```

generateSWコマンドはそもそもbuildした成果物を検索してファイルパス・リビジョンを取得するので、build後に自動実行されるnpm scriptの方が都合が良いと思われます。

では、npm scriptでビルドしましょう。

```javascript
> workbox generateSW workbox-config.js

Using configuration from /Users/tree/go/src/tree-maps-go/workbox-config.js.
The service worker was written to assets/sw.js
86 files will be precached, totalling 3.7 MB.
```

これで、assets/sw.jsが出力されます。

<a id="workbox-generateSWで生成されたswjs"></a>

##### workbox generateSWで生成されたsw.js

```javascript
/**
 * Welcome to your Workbox-powered service worker!
 *
 * You'll need to register this file in your web app and you should
 * disable HTTP caching for this file too.
 * See https://goo.gl/nhQhGp
 *
 * The rest of the code is auto-generated. Please don't update this file
 * directly; instead, make changes to your Workbox build configuration
 * and re-run your build process.
 * See https://goo.gl/2aRDsh
 */

importScripts("https://storage.googleapis.com/workbox-cdn/releases/3.0.1/workbox-sw.js");

/**
 * The workboxSW.precacheAndRoute() method efficiently caches and responds to
 * requests for URLs in the manifest.
 * See https://goo.gl/S9QRab
 */
self.__precacheManifest = [
  {
    "url": "images/tree-logo.png",
    "revision": "ddbe3304f962a6193261e1e4cfd8038e"
  },
  {
    "url": "js/app.bundle.js",
    "revision": "7230a550f7e7334453f09b7c27265e4e"
  },
  {
    "url": "js/vendor.bundle.js",
    "revision": "f7ef29ba3165cd09e58b277f0bac88aa"
  }
].concat(self.__precacheManifest || []);
workbox.precaching.suppressWarnings();
workbox.precaching.precacheAndRoute(self.__precacheManifest, {});
```

こんな感じのjsが自動生成されます。

revisionの部分ですが、ファイルの内容によってハッシュ値が決定され、ファイルの内容が変更された場合にハッシュ値が変わります。ファイルの内容を変更しなければ、何回workbox generateSWを実行しても同じrevisionになります。

つまり、前回と比較してハッシュ値が変わらなければServiceWorkerのキャッシュは継続使用され、ハッシュ値が変わったもののみServiceWorkerはキャッシュを破棄・再生性する挙動となります。

これは非常に良い仕組みですね。

<a id="自動生成されたswjsをhtmlで読み込む"></a>

#### 自動生成されたsw.jsをhtmlで読み込む

```html
navigator.serviceWorker.register('/sw.js').then(function(registration) {
    return navigator.serviceworker.ready;
}).catch(function(error) {
});
```

これでServiceWorkerが起動します。

処理フロー的には、↓こんな感じでしょうか。

1. htmlがsw.jsを読み込む。
2. sw.jsがworkbox-sw.jsをロードする。
3. workbox-sw.jsがprecacheする。

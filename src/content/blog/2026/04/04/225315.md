どういうことかというと、

| 種別             | ローカルLLM                                                         | OpenAI LLM                                                     |
|------------------|---------------------------------------------------------------------|----------------------------------------------------------------|
| アプリケーション | [Codex CLI](https://developers.openai.com/codex/cli)                | [Codex.app](https://openai.com/ja-JP/codex/get-started/)       |
| プロバイダー     | [LM Studio](https://lmstudio.ai/)                                   | [OpenAI](https://openai.com/ja-JP/codex/)                      |
| モデル           | [qwen/qwen3.5-35b-a3b](https://huggingface.co/Qwen/Qwen3.5-35B-A3B) | [GPT 4.5](https://openai.com/ja-JP/index/introducing-gpt-4-5/) |

これを両立させます。

何故わざわざ両立と言っているかというと、 `設定ファイルが共用だから` 一工夫が必要になるのです。

2026/04 時点でCodexの設定は `グローバルの ~/.codex/config.toml` のみで、プロジェクト内に `.codex/config.toml` を配置することができないからです。

`Codex CLI` の場合は実行時引数でコントロールできますが、Codex.appはそうもいかないので、工夫が必要です。

## LM Studio

モデルは何でもいいですが、 `Codex CLI` でまともに動く `qwen/qwen3.5-35b-a3b` を起動しておきます。

LM Studioであれば自動でWEB APIが `http://127.0.0.1:1234/v1` で公開されると思うので、デフォルト設定のままでいきます。

## Codex Configuration

`~/.codex/config.toml` を作成し、以下の内容にします。

```toml
#==================================================================
# 共通
#==================================================================
model_reasoning_effort = "high"
hide_agent_reasoning = true
personality = "pragmatic"
show_raw_agent_reasoning = false
notify = ["bash", "-lc", "say 完了"]

#==================================================================
# Codex.app
#==================================================================
model_provider = "openai"
model = "gpt-5.4"

#==================================================================
# Local LLM
#==================================================================
[model_providers.lmstudio-local]
name = "LMStudio"
base_url = "http://127.0.0.1:1234/v1"

[profiles.qwen3-5-35b]
model = "qwen/qwen3.5-35b-a3b"
model_provider = "lmstudio-local"

[profiles.qwen3-coder-30b]
model = "qwen/qwen3-coder-30b"
model_provider = "lmstudio-local"
```

## Codex CLI

`Codex CLI` + ローカルLLMで開発するプロジェクト内に `codex.sh` を作成し、以下の内容を記述します。

```sh
#!/bin/sh

codex \
  --oss \
  -m Qwen/Qwen3.5-35B-A3B \
  -a never \
  -s workspace-write \
  -c 'sandbox_workspace_write.network_access=true' \
  -c 'model_reasoning_effort="low"' \
  -c 'model_reasoning_summary="none"' \
  -c 'sandbox_mode="workspace-write"' \
  -c 'approval_policy="on-request"' \
  -c 'model_context_window=65536'
```

`~/.codex/config.toml` の `model` は `Codex.app` で使いたいモデルの設定をしておき、 `codex` コマンドの実行時引数で `Codex CLI` で使うモデルを設定します。

そうすることで、実行時引数で `~/.codex/config.toml` を上書きすることができます。

`Codex CLI` を起動するのに `codex` コマンドではなく `./codex.sh` で起動することになりますが、これで `Codex.app` と `Codex CLI` を同居できます。

## Codexのタスク完了時にmacに喋らせる

以下を設定しておくと、 `Codex.app` ・ `Codex CLI` のタスク完了時に喋ってくれます。

```toml
notify = ["bash", "-lc", "say 完了"]
```

そもそも macOS には `sayコマンド` があり、例えばターミナルで以下のコマンドを実行すると、喋らせることができます。

```sh
say 'こんにちは'
say -v Alex 'Hello, [[slnc 200]] World'
```

sayコマンドはファイル保存できたり、結構面白いです。

## qwen/qwen3.5-35b-a3b の reasoning_effort と reasoning_summary

```sh
  -c 'model_reasoning_effort="low"' \
  -c 'model_reasoning_summary="none"' \
  -c 'model_context_window=65536'
```

`codex.sh` の実行時引数を上記の値で上書きしている理由は以下の通りです（Chat GPTの回答）。

```
私のおすすめは、まず low + summary none です。
理由は、Qwen3.5-35B-A3B は thinking 前提の設計で、Qwen 側も「精密なコーディングタスク」は thinking モード寄りの設定を推しています。一方で、ローカル運用の Codex は推論量が多いほど待ち時間とトークン消費が増えるので、high 以上は重くなりやすいです。つまり、完全に切るより少し考えさせるほうが、Codex のエージェント動作にはバランスがよく、summary="none" で余計な表示・トークンを減らすのが安定しやすいです。Qwen の推奨サンプリングでも、精密な coding では thinking モード前提の設定が案内されています。

model_context_window は、最初は 65536 が無難です。
LM Studio は Codex 用に 25k 超を推奨し、Qwen は 128K 以上を勧めていますが、ローカル実機では context を広げるほどメモリ圧迫と遅延が増えます。なので実務上はこう切るのが扱いやすいです。

まずは 65536: ふつうの実装・修正・数ファイル横断ならこれがバランス良い
大きめ repo / 長いやり取り: 131072
重すぎる・応答が鈍い: 32768 まで下げる

これは Qwen の 128K 推奨と LM Studio の 25k+ 推奨の間を取った、ローカル実用寄りの運用判断です。Codex 側にも model_context_window があり、LM Studio 側の loaded model context もできるだけ同じ値に揃えるのが扱いやすいです。
```
